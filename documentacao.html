1. Introdução

O BookShelfly é um aplicativo mobile desenvolvido em Flutter que permite aos usuários explorar, ler e gerenciar uma biblioteca pessoal de livros. O aplicativo utiliza a API pública Gutendex, que fornece acesso a mais de 70.000 livros de domínio público do Projeto Gutenberg, uma das maiores bibliotecas digitais gratuitas do mundo.

O projeto foi desenvolvido seguindo os princípios da Clean Architecture e o padrão MVVM (Model-View-ViewModel), demonstrando boas práticas de desenvolvimento de software, separação de responsabilidades, testabilidade e manutenibilidade de código.

2. Tecnologias e Conceitos Aplicados

Este projeto foi desenvolvido como uma base prática para demonstrar, na prática, padrões de arquitetura, técnicas de desenvolvimento e soluções modernas utilizadas em aplicações mobile robustas. 3. Arquitetura do Projeto

O BookShelfly foi desenvolvido seguindo os princípios da Clean Architecture, que promove a separação de responsabilidades e independência entre camadas. A arquitetura é dividida em três camadas principais:

Estrutura de Camadas

  • Presentation Layer (MVVM): Interface do usuário e lógica de apresentação
  • Domain Layer: Regras de negócio e entidades puras
  • Data Layer: Acesso a dados e implementação de repositórios

4.1. Presentation Layer (Camada de Apresentação)

Esta camada é responsável pela interface do usuário e utiliza o padrão MVVM:

  • View: Widgets Flutter que compõem a interface (Pages e Widgets)
  • ViewModel: Gerencia o estado e a lógica de apresentação (BooksViewModel)
  • Model: Entidades do domínio que representam os dados

4.2. Domain Layer (Camada de Domínio)

Contém as regras de negócio puras, independentes de frameworks e bibliotecas externas:

  • Entities: Objetos de negócio (GutendexBook, etc.)
  • Use Cases: Casos de uso específicos (GetAllBooks, GetBookById, etc.)
  • Repository Contracts: Interfaces que definem contratos de repositórios

4.3. Data Layer (Camada de Dados)

Responsável pelo acesso a dados, tanto locais quanto remotos:

  • Data Sources: Fontes de dados (Remote e Local)
  • Models: Modelos de dados com serialização JSON
  • Repository Implementations: Implementação dos contratos definidos no Domain

4. Funcionalidades Implementadas

4.1. Página Inicial (Home)

  • Banner em Destaque: Exibe livros em destaque com rotação automática
  • Categoria Brasileiros: Seção dedicada a livros de autores brasileiros
  • Categoria Popular: Livros mais populares da biblioteca
  • Cache Inteligente: Carregamento imediato de conteúdo em cache

4.2. Biblioteca de Livros (Books)

  • Listagem de Livros: Visualização em grid ou lista
  • Busca: Pesquisa de livros por título, autor ou assunto
  • Detalhes do Livro: Informações completas sobre cada livro
  • Botões de Ação: "Ler Livro" e "Favoritar" com estados visuais

4.3. Leitor de Livros (Book Reader)

  • Visualização de Conteúdo: Leitura completa do livro
  • Navegação Rápida: Busca por página específica
  • Configurações de Leitura: Personalização da experiência
  • Progresso de Leitura: Salvamento automático do progresso

4.4. Biblioteca Pessoal (Glass)

  • Lista "Lendo": Livros que o usuário está lendo atualmente
  • Lista "Favoritos": Livros marcados como favoritos
  • Gerenciamento: Adição e remoção de livros das listas
  • Busca Personalizada: Filtro dentro da biblioteca pessoal

4.5. Notas (Notes)

  • Criação de Notas: Sistema de anotações pessoais
  • Organização Visual: Grid com cores variadas
  • Persistência: Armazenamento local das notas

4.6. Sistema de Cache

  • Cache de API: Reduz chamadas desnecessárias à API
  • Cache de Banner: Carregamento instantâneo do conteúdo em destaque
  • Fallback Offline: Funcionalidade mesmo sem internet
  • Invalidação Inteligente: Atualização automática quando necessário

8. Principais Conceitos Aplicados

8.1. Clean Architecture

A Clean Architecture promove a separação de responsabilidades através de camadas independentes, facilitando testes, manutenção e evolução do código.

8.2. SOLID Principles

  • Single Responsibility: Cada classe tem uma única responsabilidade
  • Open/Closed: Aberto para extensão, fechado para modificação
  • Liskov Substitution: Subtipos devem ser substituíveis por seus tipos base
  • Interface Segregation: Interfaces específicas ao invés de genéricas
  • Dependency Inversion: Depender de abstrações, não de implementações

8.3. Design Patterns

  • Repository Pattern: Abstração do acesso a dados
  • MVVM: Separação entre View, ViewModel e Model
  • Dependency Injection: Inversão de controle de dependências
  • Factory Pattern: Criação de objetos complexos
